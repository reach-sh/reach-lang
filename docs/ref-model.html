1.2 Language Model

This document describes the fundamental assumptions and concepts of Reach. First, we discuss the model of running a Reach program in Evaluation Model. Next, we discuss the details about compilation of a Reach program that are relevant to Reach programmers in Compilation Model. Finally, we discuss how Reach programs are syntactically constructed in Syntax Model.

1.2.1 Evaluation Model

Reach programs specify a decentralized application (DApp), which is a distributed computation involving many participants and utilizing one contract on one consensus network for reaching agreement on the intermediate values of the computation.

"Many" is a technical term that means "zero or more".

When the computation terminates, all participants agree on the resulting value, because they agreed on the intermediate values.

A consensus network is a network protocol with a network token, a set of accounts, a set of contracts, and a time. A network token is an opaque unit of account. A consensus network’s time is some monotonically increasing value from a total ordered set. Consensus networks support transfers of network tokens between accounts. An account is a unique identity (called an address) with a non-negative balance of network tokens. Accounts may sign values in a way that may not be repudiated or impersonated; this is called publication. Contracts are accounts with three extra capacities: they persistently store values (called the consensus state), they may receive publications, and when they receive publications, they systematically process them and may modify their consensus state, make publications, and may transfer network tokens in response to the reception. The chapter, Consensus Network Connectors, discusses which consensus networks are supported by Reach.

This description of consensus networks is an abstraction that may not be directly implemented by actual networks. For example, in UTXO-based networks, there is not typically an explicitly represent account balance ledger. However, such networks do abstractly have accounts with balances, because particular private keys represent accounts which have exclusive access to some set of network tokens which is their balance.

A participant is a logical actor which takes part in a DApp. It is associated with an account on the consensus network.

The same account may be used by multiple participants in a DApp.

It has persistently stored values, called its local state. It has a frontend which it interacts with. A frontend is an abstract actor which supports a set of functions which consume and produce values; when a participant invokes one of these functions it is referred to as interactaction.

Since DApps have an associated contract, they have an associated account. This account is assumed to be empty when the computation starts. Any network tokens transferred into the account must be removed by the DApp’s completion. This is called the token linearity property.

A DApp computation can be seen as a graph of steps with a unique first step. A step is a set of local computations by participants followed by a single consensus computation introduced via a single consensus transfer.

A local computation is executed by a single participant and is a sequence of local statements. A local statement either binds a piece of local state via a local expression, asserts a property of the local state, or interacts with the frontend. A consensus transfer is executed by a single participant (called the originator) which makes a publication of a set of public values from its local state and transfers some network tokens to the contract account. A consensus transfer specifies an alternative consensus transfer that is executed if the originator fails to make the transfer before a given time. All local state is initially private, until it is explicitly made public via a declassification.

A consensus computation is a graph of consensus statements with a unique first statement. A consensus statement is either computation that binds consensus state a via consensus expression, asserts a property of the consensus state, performs a transfer, evaluates a consensus expression to determine the tail of the consensus computation, or commits to the next step.

An expression is a functional computation that consumes some values and produces some values. An expression is called a local expression if it uses local primitive or consensus primitives, whereas it is called a consensus expression if does not use any local primitives. A primitive is a function from a set of local primitives and consensus primitives. One local primitive performs a declassification and returns a public version of a private value.

An assertion is either: a static assertion, which is an expression that logically evaluates to the boolean true; an assumption, which is an local expression that should evaluate to the boolean true if frontends behave honestly; a requirement, which is a consensus expression that should evaluate to the boolean true if participants behave honestly; or, a possibility assertion, which is an expression for which there exists some values that honest participants and frontends could submit which results in the evaluation of the expression to the boolean true. An honest participant is one that executes the steps specified by the DApp, while an honest frontend is one that only returns values which ensure that all assumptions evaluate to the boolean true.

A value is either a base value or a statically-sized array of base values. A base value is either an unsigned integer of 256 bits, a boolean, a string of bytes, or an account address.

1.2.2 Compilation Model

Reach programs cannot execute independently of a consensus network and a set of frontends. Thus, the semantics of Reach treats these components abstractly and does not specify their semantics. Therefore, the semantics of Reach cannot be effectively implemented directly in a virtual machine or interpreter. Instead, Reach programs are compiled to a particular consensus network connector and a set of participant backends which execute the computation of the particular consensus network. Connectors and backends are sound if they faithfully model the abstract semantics assumed by Reach.

During compilation, the Reach compiler automatically verifies that the token linearity property and all static assertions and possibility assertions are true whether participants and frontends are honest or not. If this cannot be statically verified, then the compilation process aborts.

Reach programs are specified via a subset of valid JavaScript syntax inside source files. The section Reach Programs describes the syntax of Reach programs in detail.
